# 06 — Credential Injection

## Dual-Strategy Overview

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     Credential Injection Strategies                      │
│                                                                         │
│  ┌───────────────────────────┐    ┌───────────────────────────────────┐ │
│  │  Strategy A: Runtime NVS  │    │  Strategy B: Build-time Kconfig   │ │
│  │                           │    │                                   │ │
│  │  1. Flash firmware first  │    │  1. Generate overlay.conf         │ │
│  │  2. Device boots with     │    │  2. Trigger CI build with overlay │ │
│  │     provisioning listener │    │  3. Firmware built with creds     │ │
│  │  3. Send PROV:* commands  │    │     baked into binary             │ │
│  │     via serial            │    │  4. Flash pre-configured binary   │ │
│  │  4. Creds stored in NVS   │    │                                   │ │
│  │                           │    │  ✓ No serial protocol needed      │ │
│  │  ✓ Update without reflash │    │  ✓ Works offline after flash      │ │
│  │  ✓ Per-device unique IDs  │    │  ✗ Requires rebuild per config    │ │
│  │  ✗ Needs serial connection│    │  ✗ Creds in binary (less secure)  │ │
│  └───────────────────────────┘    └───────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────────┘
```

## Strategy A: PROV:* Serial Protocol

```
Web UI (CredentialWriter)          ESP32 (Zephyr App)
        │                                 │
        │  PROV:START\n                    │
        │ ───────────────────────────────► │
        │                                 │ Initialize NVS
        │  PROV:OK:READY\n                │
        │ ◄─────────────────────────────── │
        │                                 │
        │  PROV:SET:wifi_ssid=MyNet\n     │
        │ ───────────────────────────────► │
        │                                 │ nvs_write("wifi_ssid","MyNet")
        │  PROV:OK:SET\n                  │
        │ ◄─────────────────────────────── │
        │                                 │
        │  PROV:SET:wifi_password=***\n   │
        │ ───────────────────────────────► │
        │                                 │ nvs_write("wifi_password","***")
        │  PROV:OK:SET\n                  │
        │ ◄─────────────────────────────── │
        │                                 │
        │  PROV:CERT:tls_ca=base64...\n   │
        │ ───────────────────────────────► │
        │                                 │ Store PEM in NVS large entry
        │  PROV:OK:CERT\n                 │
        │ ◄─────────────────────────────── │
        │                                 │
        │  PROV:COMMIT\n                  │
        │ ───────────────────────────────► │
        │                                 │ NVS commit + validate
        │  PROV:OK:COMMIT\n               │
        │ ◄─────────────────────────────── │
        │                                 │
        │  PROV:STATUS\n                  │
        │ ───────────────────────────────► │
        │                                 │
        │  PROV:STATUS:wifi=OK,mqtt=OK\n  │
        │ ◄─────────────────────────────── │
```

## Protocol Commands

| Command | Direction | Description |
|---------|----------|-------------|
| `PROV:START` | Host → Device | Begin provisioning session |
| `PROV:SET:key=value` | Host → Device | Set key-value credential |
| `PROV:CERT:name=base64` | Host → Device | Set certificate (base64-encoded PEM) |
| `PROV:COMMIT` | Host → Device | Commit all changes to NVS |
| `PROV:STATUS` | Host → Device | Query provisioning status |
| `PROV:OK:*` | Device → Host | Success response |
| `PROV:ERR:message` | Device → Host | Error response |

## Strategy B: Kconfig Overlay

```
# overlay.conf — Generated by Credential Editor
CONFIG_WIFI_SSID="MyNetwork"
CONFIG_WIFI_PASSWORD="secret123"
CONFIG_MQTT_BROKER="mqtt://broker.local:1883"
CONFIG_MQTT_USER="device01"
CONFIG_DEVICE_ID="esp32-sensor-001"
```

This file is sent to GitHub Actions via `workflow_dispatch` inputs
and applied during the Zephyr build process.
